# Website Build System

This skill handles all build chores for the site. Follow the following procedure exactly.
If you are working with multiple pages, you should repeat this procedure for each page.

## Build Page

Identify the page you want to release from the conversation history. This will be a markdown file with frontmatter.

This repository uses a custom static site generator that converts `.md` files to `.html` files. The builder is designed to be idempotent and efficient, only rebuilding pages that have changed.

In most cases, you may build the page by running the builder on its `.md` file from the workspace root:

```
source venv/bin/activate && python builder/build.py <path-to-md-file>
```

If the page's frontmatter contains `manual_html: true`, you must verify the `.html` file directly instead of running the builder. Do NOT delete or rebuild `manual_html: true` pages. These are pages with custom interactive content that cannot be auto-generated.

If there are changes to the `.html` file, commit them with the message `chore: rebuild <page-name> page`.

## Build Parent Index

Identify the parent index page that links to the page you just built. Make sure the parent index page is updated to link to the new page.
If the parent index page does not have a link, AskUserQuestion to find an appropriate tagline and placement for the new page. Then, add the link to the parent index page and run the builder on the parent:

```
source venv/bin/activate && python builder/build.py <path-to-parent-index-md-file>
```

If there are changes to the markdown or HTML of the parent index page, commit them with the message `chore: add listing for <page-name>`.

## Add to Site Map

This website has a custom interactive site map that lists all pages. This must include the new page to ensure is discoverable.

Do NOT manually edit any files yet. After building the page and its parent index, run the site map generator to update the site map.

```
source venv/bin/activate && python map/generate.py
```

Verify that the ONLY changes are:
- `map/index.md`: exactly one new line for the new page in the ASCII tree
- `map/map.json`: exactly one new entry for the new page

The new JSON entry for the page should look like this:

```
  {
    "name": "My New Page",
    "url": "/path/to/parent-index/new-page",
    "parent": "/path/to/parent-index#subsection",
    "display": true
  },
```

If necessary, you may now manually edit the JSON entry using the guidelines below for each field.

### Name Field

The `name` field is used to display the page in the site map. It should be a terse title that matches the page's content. It need not stand alone: it is only used in the context of the parent index page, so it can be more concise than the full page title. For example, if the page is "The Ultimate Guide to X", and it is linked from a subsection of the parent index all about X, the `name` field can simply be "Guide".

### Parent Field

The `parent` field specifies the parent page in the site map. It should be a relative URL to the parent index page: in particular, it is the "url" field of the parent page's JSON entry, with an optional `#subsection` suffix if needed.

The `#subsection` is not generated by default. This is an optional feature which splits the interactive map tree visually for clarity. It is only needed if the page is linked from a subsection of the parent index. If the page is linked from the top level of the parent index, or there are very few sibling pages, you can leave the path alone. You should check the existing entries of sibling pages in `map/map.json` for examples of when to use the `#subsection` field. If you are unsure, ask for clarification.

### Display Field

If the page should not be listed in the site map for any reason, set `"display": false` in the JSON entry. This will hide the page from the site map but keep it accessible through direct links. This is only appropriate for niche pages that are not relevant to most users and would clutter the site map. For example, if a page containing a puzzle has the solution on a separate page, the solution page can be hidden from the site map since it is only relevant to users who have already engaged with the puzzle. When in doubt, ask for clarification on whether a page should be hidden from the site map.

When you are ready, run the generator again.

```
source venv/bin/activate && python map/generate.py
```

Commit your changes to the site map with the message `chore: add <page-name> to site map`.

## Final Checks

To check your work, run the following steps.

### Verify Links

Check that each new page is linked from its parent index page. Also check that the site map's JSON file includes the new page with the correct name and parent. This allows the new page to be discoverable through navigation and the site map.

### Verify Build

Run the builder without arguments to perform a full build of the site:

```
source venv/bin/activate && python builder/build.py
```

Ensure that the build completes without errors and that there are no new changes to commit after the build.

### Verify Map

Run the site map generator again to verify that it has been built correctly.

```
source venv/bin/activate && python map/generate.py
```

Ensure that there are no new changes to commit after the site map generation.

## Merging into Main

After verifying that everything is correct, merge your changes to main. Do NOT merge to main yourself. The user will handle merging to main after you have completed all build steps and verified that there are no errors or changes to commit.
